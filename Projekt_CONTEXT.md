# Screentation V2 — Документация Контекста Проекта

Этот документ предназначен для архитекторов и разработчиков. Он описывает внутреннюю архитектуру, ключевые технические решения, исправленные критические ошибки с их костылями (workarounds), а также вектор развития проекта.

---

## 1. Суть проекта и Стек

**Screentation** — десктопное Windows-приложение для мгновенного захвата скриншотов из буфера обмена, их разметки с помощью векторных аннотаций (шаги, текст, стрелки, рамки, размытие, штамп-стёрка) и пакетного экспорта. Приложение ориентировано на быстрое создание пошаговых инструкций и документации по интерфейсам.

### Технологический стек

* **Среда выполнения**: .NET 10.0 (TargetFramework: `net10.0-windows10.0.26100.0`, минимальная поддерживаемая версия Windows: 10.0.17763.0)
* **Язык программирования**: C# 13
* **Интерфейс (UI)**: WinUI 3 (входит в состав Windows App SDK версии `2.2.0`)
* **Тип сборки**: Unpackaged (`<WindowsPackageType>None</WindowsPackageType>`), Self-Contained (автономный `.exe` со всеми зависимостями)
* **Графика и рендеринг**: Microsoft.Graphics.Win2D версии `1.4.0` (аппаратное ускорение рендеринга аннотаций на холсте)
* **Системный трей**: H.NotifyIcon.WinUI версии `2.4.1` (реализация значка в трее и контекстного меню)
* **Инструменты сборки**: Microsoft.Windows.SDK.BuildTools `10.0.28000.1839`, Microsoft.Windows.SDK.BuildTools.WinApp `0.3.2`
* **База данных**: Отсутствует. Настройки приложения сохраняются локально в JSON-формате (`settings.json`) в папку `AppData/Local/Screentation` с использованием `System.Text.Json`.

## 2. Архитектура и Паттерны

### Структура папок и файлов проекта

```
Screentation/
├── Assets/                 # Иконки, заставки и графические ресурсы
├── Properties/             # Профили сборки и публикации
├── AnnotationCanvas.cs     # Логика интерактивного холста (события мыши, скейлинг, инструменты)
├── AnnotationDrawer.cs     # Рендеринг векторных примитивов через Win2D
├── ClipboardMonitor.cs     # Низкоуровневый перехват буфера обмена через Win32 API
├── ExportManager.cs        # Пакетный рендеринг высокого разрешения и экспорт на диск
├── HistoryManager.cs       # Стек Undo/Redo для каждой сессии скриншотов
├── ImageHelper.cs          # Масштабирование и конвертация SoftwareBitmap
├── SettingsManager.cs      # Загрузка/сохранение пользовательских настроек в JSON
├── Models.cs               # Доменные модели аннотаций (Rect, Arrow, Step, Text, Blur, Eraser)
├── MainWindow.xaml/.cs     # Главный контейнер приложения, управление треем и заголовком
├── MainPage.xaml/.cs       # Основной экран приложения (список сессий, панель настроек, холст)
├── app.manifest            # Манифест приложения с запросом прав и DPI-соответствия
└── Screentation.csproj     # Конфигурация сборки MSBuild
```

### Архитектурные паттерны

1. **Разделение слоев (Separation of Concerns)**:
   * **Слой моделей (Domain Models)**: Все элементы аннотаций унаследованы от абстрактного класса `AnnotationElement` (`Models.cs`). Они хранят исключительно геометрические параметры и цвет в исходных координатах изображения (в пикселях).
   * **Слой рендеринга (Rendering Engine)**: `AnnotationDrawer` содержит только чистые статические методы рисования примитивов на контексте `CanvasDrawingSession`. Он ничего не знает о состоянии мыши, выделении или масштабе.
   * **Слой управления вводом (UI Controller)**: `AnnotationCanvas` управляет масштабом холста, смещением панорамирования, преобразует экранные координаты в оригинальные и обратно, а также создаёт интерактивные элементы UI (например, `TextBox` для ввода текста).
2. **State Management (Управление состоянием)**:
   * Текущий активный скриншот и все его аннотации инкапсулированы в класс `ScreenshotSession`.
   * Изменение списка аннотаций отслеживается через `ObservableCollection<AnnotationElement>`, на которое подписывается холст для вызова перерисовки (`Invalidate()`).
   * Векторные аннотации хранятся в **оригинальных координатах скриншота**. Любые экранные трансформации (Zoom & Pan) применяются глобально к матрице трансформации Win2D перед отрисовкой:
     ```csharp
     ds.Transform = Matrix3x2.CreateScale(_scale) * Matrix3x2.CreateTranslation(_offsetX, _offsetY);
     ```

## 3. Реализованная логика (Current State)

* **Фоновый мониторинг буфера обмена**: Приложение регистрирует окно как слушатель буфера обмена (`AddClipboardFormatListener`). При сворачивании в трей перехват скриншотов продолжает работать стабильно без потери фокуса.
* **Интерактивные векторные инструменты**:
  * *Rect (Прямоугольник)*: Поддержка контура или полупрозрачной заливки.
  * *Arrow (Стрелка)*: Направление отрисовывается динамически на основе точек начала и конца.
  * *Blur (Размытие)*: Применяет эффект `GaussianBlurEffect` Win2D к исходному изображению в выбранной области.
  * *Eraser (Штамп-стёрка)*: Копирует пиксели из области-источника и переносит их на целевую область для маскировки конфиденциальных данных.
  * *Step (Шаги)*: Автоинкрементный маркер шагов. Поддерживает форматы: `Numeric` (`1, 2, 3`), `AlphaUpper` (`A, B... Z, AA`) и `AlphaLower` (`a, b... z, aa`).
  * *Text (Текстовые аннотации)*: Размещение текста с кастомным размером шрифта.
* **Zoom & Pan холста**: Масштабирование через `Ctrl + Wheel` (центруется относительно курсора мыши), панорамирование средней кнопкой мыши и двухсторонняя синхронизация со слайдером масштаба в UI.
* **Кадрирование**: Обрезка исходного изображения с помощью рамки с последующим автоматическим пересчетом координат и сдвигом всех существующих аннотаций.
* **Настройки**: Полный спектр выбора цвета через WinUI `ColorPicker`. Выбранный цвет, масштаб по умолчанию и путь экспорта сохраняются и загружаются автоматически.

## 4. Ключевые решения и Костыли (Context & Workarounds)

> [!IMPORTANT]
> Изложенные ниже решения были применены для обхода ограничений операционной системы Windows и WinUI 3. Модификация этих участков кода без чёткого понимания причин приведет к немедленной поломке ключевого функционала приложения.

### 4.1. Обход ограничений WinRT Clipboard (Фоновый захват)
* **Проблема**: Использование стандартного WinRT `Clipboard.GetContent()` в фоновом режиме (когда приложение свернуто) выбрасывает исключение отсутствия фокуса (Access Denied / Focus Exception).
* **Костыль (Решение)**: Использование низкоуровневого Win32 API для прямого чтения DIB-структуры из буфера обмена.
```csharp
// Из файла ClipboardMonitor.cs
if (!OpenClipboard(IntPtr.Zero)) return (null, false);
try {
    IntPtr hDIB = GetClipboardData(CF_DIB); // Формат Device Independent Bitmap
    if (hDIB == IntPtr.Zero) return (null, false);
    IntPtr pDIB = GlobalLock(hDIB);
    // ...Парсинг BITMAPINFOHEADER и копирование байтов пикселей...
} finally {
    CloseClipboard();
}
```

### 4.2. Обработка гонки за буфер обмена (Race Conditions)
* **Проблема**: Системный инструмент создания скриншотов (например, «Ножницы» Windows) блокирует буфер обмена на короткое время после отправки события `WM_CLIPBOARDUPDATE`. Немедленное чтение возвращает ошибку блокировки.
* **Костыль (Решение)**: Внедрена 100-миллисекундная начальная задержка перед чтением и цикл из 10 попыток повторного открытия буфера обмена с интервалом 50 мс.
```csharp
// Из файла ClipboardMonitor.cs (ProcessClipboardAsync)
await Task.Delay(100); // Первичная пауза
for (int i = 0; i < 10; i++) {
    var result = GetClipboardImageWin32();
    if (result.isDuplicate) break;
    bitmap = result.bitmap;
    if (bitmap != null) break;
    await Task.Delay(50); // Повторная попытка через 50мс
}
```

### 4.3. Умное подавление дубликатов с временным окном
* **Проблема**: Некоторые программы перезаписывают буфер обмена дважды за одну операцию копирования, что приводит к созданию дубликатов скриншотов. В то же время пользователю может быть необходимо сделать два одинаковых скриншота подряд с интервалом в несколько секунд для нанесения разных аннотаций.
* **Костыль (Решение)**: При получении кадра снимается отпечаток (разрешение + 10 sample-пикселей в ключевых точках экрана). Если кадры совпадают, они отсекаются как дубликаты **только в том случае**, если интервал между ними составляет менее 2.0 секунд.
```csharp
// Из файла ClipboardMonitor.cs
if (isDuplicate && (DateTime.UtcNow - _lastCapturedTime < TimeSpan.FromSeconds(2.0))) {
    return (null, true); // Пропуск дубликата
}
```

### 4.4. Стабильное позиционирование текстового редактора (TextBox)
* **Проблема**: Стандартный текстовый ввод поверх холста Win2D не поддерживается напрямую. Ранние реализации TextBox позиционировались случайным образом и перекрывали элементы интерфейса из-за некорректного учета масштаба.
* **Костыль (Решение)**: Контрол `TextBox` создается динамически и добавляется на холст `AnnotationCanvas` (наследник `Grid`). Его координаты пересчитываются из оригинальных в экранные с учетом текущего скейлинга (`_scale`), а размеры синхронизируются с размером шрифта.
```csharp
// Из файла AnnotationCanvas.cs (OpenTextEditor)
Point screenPt = TransformPointToScreen(element.Position);
_activeTextEditor.RenderTransform = new TranslateTransform {
    X = screenPt.X,
    Y = screenPt.Y
};
_activeTextEditor.FontSize = element.FontSize * _scale;
```

### 4.5. Сдвиг аннотаций при кадрировании (Crop Coordinates Shifting)
* **Проблема**: При обрезке скриншота изображение уменьшается, из-за чего все нарисованные векторные элементы смещаются относительно фона, так как их координаты привязаны к старому левому верхнему углу (0,0).
* **Костыль (Решение)**: При подтверждении обрезки вычисляется вектор смещения обрезки и все элементы принудительно сдвигаются в обратную сторону.
```csharp
// Из файла AnnotationCanvas.cs (ApplyCrop)
Vector2 shift = new Vector2(-(float)cropRect.X, -(float)cropRect.Y);
foreach (var element in Session.Annotations) {
    element.Move(shift);
}
```

## 5. Требования к окружению (Environment)

* **Переменные окружения (`.env`)**: Не требуются. Приложение полностью автономно и не взаимодействует с внешними веб-сервисами.
* **Внешние службы**: Не требуются (база данных отсутствует, Docker не нужен).
* **Сборка и Запуск**:
  * Локальный запуск (Debug-режим):
    ```bash
    dotnet run --project Screentation/Screentation.csproj
    ```
  * Сборка и публикация автономного (Self-Contained) релиза:
    ```bash
    dotnet publish Screentation/Screentation.csproj -c Release -r win-x64 --self-contained true
    ```
    *Результат сборки будет помещен в:* `Screentation/bin/Release/net10.0-windows10.0.26100.0/win-x64/publish/`

## 6. Технический долг и Известные проблемы

1. **Зависимость экспорта WebP от системных кодеков Windows**:
   * *Проблема*: Метод сохранения WebP в `ExportManager.cs` задействует WIC-энкодер. Если в ОС Windows не установлены расширения изображений WebP, экспорт завершится ошибкой.
   * *Решение в коде*: Внедрен fallback-перехватчик, автоматически сохраняющий изображение в формате `.png` вместо выброса фатального исключения. Требуется реализовать UI-уведомление пользователя об автозамене формата.
2. **Приближенный хит-тест текстовых блоков**:
   * *Проблема*: Выделение текста кликом мыши (`TextElement.HitTest`) рассчитывает габариты строки грубо на основе ширины символа (`FontSize * 0.6f`). На некоторых шрифтах это может приводить к смещению рамки выделения.
   * *Рефакторинг*: В будущем заменить на точный замер габаритов строки с использованием `CanvasTextLayout` из Win2D.
3. **Утечки SoftwareBitmap при интенсивном использовании**:
   * *Проблема*: Объекты `SoftwareBitmap` выделяют неуправляемую память. При длительной работе приложения без перезапуска сборщик мусора может не успевать вовремя очищать ресурсы. Требуется провести профилирование утечек памяти в долгосрочной перспективе.
